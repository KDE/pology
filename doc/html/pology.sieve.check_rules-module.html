Module check_rules

Try to fail messages by rules and warn when that happens.

This sieve applies a collection of special rules to messages, reporting whenever a rule "fails" a message -- rules are usually written to detect messages faulty, or possibly such, in a certain sense.

By default, the sieve reads rules from Pology's internal l10n/<lang>/rules/ directories, i.e. written for specific languages, and possibly specific translation environments within a given language. Read about how to write rules and create rule files in the misc.rules module documentation.

The sieve parameters are:

  • lang:<language>: language for which to fetch and apply the rules
  • env:<environment>: specific environment within the given language; if not given, only environment-agnostic rules are applied
  • envonly: when a specific environment is given, apply only the rules explicitly belonging to it (ignoring environment-agnostic ones)
  • rule: comma-separated list of specific rules to apply, containing rule identifiers; if not given, all rules for given language and environment are applied
  • stat: show statistics of rule matching at the end
  • accel:<characters>: characters to consider as accelerator markers
  • markup:<mkeywords>: markup types by keyword (comma separated)
  • xml:<filename>: output results of the run in XML format file
  • rfile:<filename>: read rules from this file, instead of from Pology's internal rule files
  • showfmsg: show filtered message too when a rule fails a message
  • nomsg: do not show message content, only problem descriptions

Parameters accel and markup set accelerator markers (e.g. _, &, etc.) and markup types by keyword (e.g. xml, qtrich, etc.) that may be present in sieved catalogs. However, providing this information by itself does nothing, it is only forced on catalogs (overriding what their headers state, if anything) such that filter and validation hooks can properly process messages. See documentation to rules for setting up these in rule files.

Certain rules may be selectively disabled on a given message, by listing their identifiers (id= rule property) in skip-rule: embedded list:

   # skip-rule: ruleid1, ruleid1, ...
